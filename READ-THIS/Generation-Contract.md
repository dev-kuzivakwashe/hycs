# Generation Contract

## Purpose

The Generation Contract defines what HYCS is allowed to generate.

This document protects the experiment.

Contributors may improve how HYCS generates applications, but they should not unintentionally change what HYCS is attempting to generate.

Changes to this contract require discussion before implementation.

---

# Core Principle

HYCS exists to test whether AI can generate and evolve useful applications using no-framework plain code.

The output generated by HYCS must remain:

* Portable
* Understandable
* Editable
* Deployable
* Independent

The generated application should belong to the user, not the platform.

---

# Output Philosophy

HYCS should generate applications that can be:

* Opened directly in a browser
* Edited without special tooling
* Deployed without build systems
* Shared without framework dependencies
* Maintained without framework dependencies

The generated application should not require HYCS to function after export.

---

# Output Technologies

## Required

HYCS-generated applications may use:

* HTML
* CSS
* JavaScript
* Standard Browser APIs
* Standard Web APIs

---

## Prohibited

HYCS-generated applications may not use:

* React
* Vue
* Angular
* Svelte
* Next.js
* Nuxt
* Remix
* Framework runtimes
* Bundler-required output
* Framework-dependent builds

If a generated application requires a framework, build process or framework runtime to function, it violates this contract.

---

# Deployment Principle

Generated applications should be deployable as-is.

Examples include:

* Local machine
* Static hosting
* GitHub Pages
* Cloudflare Pages
* Netlify
* Vercel static deployments
* Traditional web servers

The deployment process should remain simple.

---

# Structure Principle

HYCS should favor simplicity over abstraction.

Generated applications should remain understandable when opened directly by a developer.

The generated output should avoid unnecessary architectural complexity.

---

# HTML Requirements

Generated HTML should:

* Be semantic
* Be accessible
* Use meaningful structure
* Use appropriate elements
* Avoid unnecessary nesting

Generated HTML should prioritize clarity.

---

# CSS Requirements

Generated CSS should:

* Be readable
* Be maintainable
* Be responsive
* Be mobile-friendly
* Support accessibility

HYCS may establish design-system rules to improve consistency across generated applications.

Design systems do not violate the no-framework principle.

---

# JavaScript Requirements

Generated JavaScript should:

* Use standard browser APIs
* Use standard DOM APIs
* Use Fetch API when appropriate
* Use browser-supported functionality

Generated JavaScript should remain understandable without framework knowledge.

---

# Design Principle

Generated applications should aim for:

* Professional appearance
* Accessibility
* Responsiveness
* Consistency
* Usability

The objective is not a vibe-coding platform clone.

The objective is a vibe-coded output with 0 frameworks involved.

---

# Libraries

Libraries and frameworks are not the same thing.

HYCS may use lightweight libraries within generated output if they improve consistency, usability or maintainability without introducing framework dependency.

Examples may include:

* Typography libraries
* Icon libraries
* Utility libraries
* Accessibility libraries

Libraries should support the generated application.

Libraries should not become a hidden framework.

Contributors proposing library usage should clearly explain:

* Why it is needed
* What problem it solves
* Why a simpler alternative is insufficient

---

# User Ownership

Users should own the generated application.

HYCS should not create unnecessary dependencies between the generated application and the HYCS platform.

After export, the application should remain functional without HYCS.

---

# Export Principle

Exported applications should remain:

* Portable
* Editable
* Deployable
* Independent

Export should not lock users into the HYCS ecosystem.

---

# Evaluation Criteria

Contributors should evaluate generated output using questions such as:

* Is the application useful?
* Is the application understandable?
* Is the application deployable?
* Is the application responsive?
* Is the application accessible?
* Is the application portable?
* Does the application remain independent?

---

# Future Evolution

The Generation Contract may evolve as the experiment evolves.

However, the following principles are considered protected:

* No-framework output
* HTML output
* CSS output
* JavaScript output
* User ownership
* Output portability
* Output independence

Changes affecting these principles require discussion and maintainer approval.

---

# Decision Rule

When uncertainty exists, ask:

> Does this change help HYCS generate better applications without violating the no-framework principle?

If yes, continue.

If not, reconsider the change.
