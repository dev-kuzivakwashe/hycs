# GitHub Deployment Specification

> Implementation status: v1 shipped using Personal Access Token. OAuth listed as a future enhancement.

## Purpose

GitHub Deployment lets users store, manage and own applications generated by HYCS. It supports the core HYCS principle of user ownership: generated applications should never be trapped inside HYCS.

```
Prompt -> Generation -> Deploy To GitHub -> User Repository -> Permanent Ownership
```

Generated applications remain accessible even if the user never returns to HYCS.

## Core Principle

HYCS generates. Users own. GitHub Deployment makes that ownership practical.

## Authentication (v1)

v1 ships **Personal Access Token (PAT)** authentication, not OAuth.

- The user creates a token at `github.com/settings/tokens` with `repo` scope (or `public_repo` for public-only).
- The token is saved in Settings -> Connections -> GitHub, or pasted directly in the deploy dialog.
- HYCS sends the token only as the `Authorization: Bearer` header to `api.github.com` from a server function. It is never logged, never written to chat history and never committed.

OAuth is a future enhancement. PAT was chosen for v1 because OAuth requires a registered GitHub App, a callback route and a token-exchange flow, while PAT meets every acceptance criterion in the spec today.

## Deployment Modes

### Mode 1 - Create Repository

User picks `Create new repo`, names the repo, picks public or private, hits Deploy. HYCS calls `POST /user/repos` then pushes the generated files into the new repo via the Git Data API.

### Mode 2 - Existing Repository

User picks `Existing repo` and enters either `name` (assumed to be under their user) or `owner/name`. HYCS validates access with `GET /repos/{owner}/{repo}` and then pushes new files as a single commit on the default branch.

## Repository Structure

```
project-name/
  index.html
  about.html
  ...
  shared/
    header.html
    footer.html
    theme.css
  README.md
```

Each page is a complete standalone HTML document (header, footer and theme stitched in). The structure remains understandable without HYCS.

## Push Mechanism

HYCS uses the Git Data API for atomic, single-commit pushes:

1. `GET /repos/{r}/git/ref/heads/{branch}` - current head sha.
2. `POST /repos/{r}/git/blobs` (per file) - base64 content blob.
3. `POST /repos/{r}/git/trees` with `base_tree` - new tree referencing the blobs.
4. `POST /repos/{r}/git/commits` - commit message + parent sha + tree sha.
5. `PATCH /repos/{r}/git/refs/heads/{branch}` - fast-forward the ref.

This means every deploy is a single, clean commit.

## Commit Strategy

Each deployment creates a single meaningful commit. Default message: `Initial project generation`. The user can edit the message before each deploy. The UI suggests update-style messages (`Updated landing page design`) after the first push.

## Deployment Metadata

HYCS writes a `README.md` into the generated repo containing the project brand, the planner-assigned title, the generation date and the export date. It does not introduce any HYCS runtime dependency.

## Deployment Status

The dialog walks through: `Not Connected -> Connected -> Deploying -> Success` or `Failed`. Success shows the repository link and the GitHub Pages URL the user would need to enable. Failure shows the spec's plain-language messages and offers a retry.

## Error Handling

| GitHub response | HYCS message |
| --- | --- |
| 422 "name already exists" | A repository with this name already exists. Choose another name or use the existing repository option. |
| 401 Unauthorized | GitHub authentication could not be completed. Please reconnect your account and try again. |
| 403 Forbidden | HYCS does not have permission to access this repository. |
| 404 Not Found | Repository not found. Check the name and try again. |
| Network failure | Deployment could not be completed because the connection was interrupted. |

## GitHub Pages

v1 prepares the repository for GitHub Pages: every generated page is static, paths are root-relative, and the structure is Pages-compatible. The user enables Pages in the repository Settings. Automatic Pages activation is a future enhancement.

## Security

HYCS will never commit:

- API keys (LOVABLE_API_KEY, BYOK keys, Pexels, Pixabay)
- Local HYCS settings (`hycs:settings:v1`, `hycs:byok:v1`)
- Project metadata snapshots
- The GitHub token itself

The deploy file list is built explicitly from the project state, not from a file-system walk.

## Export Compatibility

The dialog keeps `Download zip` next to `Deploy`. Users can always export without GitHub. GitHub deploy is an additional ownership path, not a required one.

## Ownership

After deploy:

- The repository belongs to the user.
- HYCS does not require continued access for the repo to function.
- The user can clone, modify, deploy elsewhere or revoke the HYCS token without breaking the project.

## Acceptance Criteria

All met in v1:

- Connect GitHub via PAT.
- Create a brand new repository.
- Deploy generated files as a single clean commit.
- Redeploy updates into the same repository.
- Repository ownership stays with the user.
- No secrets are committed.
- Deployment status is visible and understandable.
- Invalid token / missing repo / name collision all produce the spec's plain-language messages.
- Zip export remains available without GitHub.

## Future

Real GitHub OAuth (App registration + callback), automatic GitHub Pages activation, custom domains, GitLab + Bitbucket support, branch selection, commit history viewer, deployment rollback.

## Decision Rule

> Does this feature improve project ownership without creating dependency on HYCS?

If yes, ship. If not, simplify.
