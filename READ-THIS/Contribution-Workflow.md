# Contribution Workflow

## Purpose

HYCS is an experiment.

Contributions should help us better understand, improve or validate the experiment.

The goal is not to accumulate code.

The goal is to solve meaningful problems while preserving the purpose of the project.

Before contributing, contributors should read:

* HYCS Governance.md
* Generation Contract.md
* README.md

---

# Contributor Types

HYCS welcomes contributions from different disciplines.

Not every contribution requires code.

---

## Developers

Developer contributions should primarily improve HYCS-generated output, generation quality, deployment systems, integrations and overall system capabilities.

Examples:

* Agent improvements
* Generation quality improvements
* Export systems
* Deployment systems
* GitHub integration
* Model integration
* Infrastructure improvements
* Performance improvements

The HYCS platform interface is a protected area.

Developers should not redesign, restructure or remove core interface elements unless the change is required by an approved feature or bug fix.

Major interface changes require maintainer approval before implementation.

---

## Researchers

Research contributions should primarily improve generated output, generation quality, deployment, integrations and experiment validation.

Examples:

* Benchmarking
* Model evaluations
* Architecture proposals
* Experiments
* Technical investigations

Research that changes the purpose, philosophy or protected areas of HYCS requires discussion before implementation.

---

## Designers

Design contributions should primarily improve HYCS-generated outputs, not the HYCS platform interface.

Designers may contribute:

* Generated-site design systems
* Layout rules for generated pages
* Typography systems for generated output
* Colour systems for generated output
* Accessibility improvements for generated output
* Mobile and responsive design standards
* Visual consistency rules for generated applications

The HYCS platform UI is a protected area.

Contributors should not redesign, remove or restructure the HYCS interface unless the change is tied to a specific approved feature.

UI removal or major interface changes require authoritative maintainer confirmation before implementation.

---

## Testers

Testing should primarily focus on generated websites and applications.

Examples:

* Responsiveness
* Accessibility
* Cross-browser support
* Generation consistency
* Export quality
* Deployment validation
* Bug reports
* Reproduction steps
* Validation testing
* Cross-device testing

Platform testing is valuable but should not become the primary focus of testing efforts.

---

## Technical Writers

Contribute:

* Documentation
* Guides
* Tutorials
* Architecture notes
* Research summaries
* Contributor documentation
* Integration documentation

Documentation should help future contributors understand not only what exists, but why it exists.

---

# HYCS Platform Team

The HYCS platform itself is maintained by a smaller group of contributors responsible for:

* Core interface decisions
* Architecture decisions
* Governance
* Protected areas
* Release management
* Platform direction

Most contributors are expected to improve generated output, research, integrations and supporting systems rather than redesigning the platform itself.

---

# Contribution Lifecycle

Every contribution should follow the same process.

Problem
→ Discussion
→ Solution Proposal
→ Approval
→ Implementation
→ Documentation
→ Review
→ Merge

The objective is clarity before code.

---

# Step 1 — Identify A Problem

Every contribution begins with a problem.

Avoid beginning with:

> I want to build this.

Begin with:

> This problem exists.

Examples:

* Generated pages are inconsistent.
* GitHub deployment is missing.
* Prompt planning needs improvement.
* Mobile output is unreliable.
* Generated designs lack consistency.

---

# Step 2 — Open Or Join An Issue

If an issue already exists:

Join the discussion.

If an issue does not exist:

Create one.

Every issue should answer:

## Problem

What is wrong?

---

## Why It Matters

Why should the project care?

---

## Expected Result

What should improve?

---

## Protected Areas

What must not change?

---

## Testing Method

How will success be measured?

---

# Step 3 — Propose A Solution

Before implementation, explain:

* What will change
* Why this approach was selected
* Alternatives considered
* Tradeoffs introduced

Major changes should be discussed before implementation begins.

---

# Step 4 — Build

Implement the solution.

Contributors should aim for:

* Simplicity
* Clarity
* Consistency
* Compatibility with HYCS Governance

Avoid unnecessary complexity.

Avoid introducing solutions that create larger problems.

---

# Step 5 — Document

Documentation is required.

Code explains implementation.

Documentation explains intent.

Every meaningful contribution should leave documentation behind.

Examples:

* Research findings
* Design decisions
* Benchmark results
* Setup guides
* Integration notes
* Testing results

Code without documentation should be considered incomplete.

---

# Step 6 — Submit A Pull Request

Every pull request should answer:

## What Changed?

Describe the change.

---

## Why?

Explain the reason.

---

## What Files Were Modified?

List major areas affected.

---

## How Was It Tested?

Describe validation steps.

---

## Documentation Added?

Link relevant documentation.

Pull requests without documentation may be rejected until documentation is provided.

---

# Step 7 — Review

Reviewers should evaluate:

* Problem solved?
* Governance respected?
* Documentation provided?
* Testing completed?
* Experiment preserved?

The goal is improvement, not gatekeeping.

---

# Contributor Recognition

HYCS recognizes meaningful contributions.

Contributors may receive:

* Contributor profile pages
* Repository recognition
* Documentation attribution
* Research attribution
* LinkedIn references
* GitHub references

Contribution quality matters more than contribution volume.

---

# Contributor Profiles

Contributors may be assigned a profile page.

Example:

/contributors/jane-doe

Contributor profiles may include:

* Name
* Biography
* LinkedIn
* GitHub
* Areas of contribution
* Research
* Documentation
* Pull requests
* Notable achievements

The objective is to create a visible record of meaningful work and meaningful thinking.

---

# Documentation Standard

Future contributors should be able to understand:

* What was built
* Why it was built
* Why alternative approaches were rejected
* What was learned

Documentation should reduce repeated discussions and prevent knowledge loss.

---

# Decision Rule

When uncertainty exists, ask:

> Does this contribution help solve a meaningful problem while preserving the HYCS experiment?

If yes, continue.

If not, reconsider the contribution.
