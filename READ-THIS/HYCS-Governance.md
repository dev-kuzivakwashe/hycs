# HYCS Governance

## Purpose

Hyper-text Cascading Scripts (HYCS) is an open-source experiment testing whether AI can generate and evolve useful applications with no-framework plain code.

The output generated by HYCS must be HTML, CSS and JavaScript.

Frameworks like React, Vue and Angular are strictly prohibited in HYCS output.

Developers are allowed to use frameworks to develop HYCS, but HYCS is not allowed to use frameworks to develop sites.

That distinction matters.

HYCS itself can use tools that help humans build the platform. The sites generated by HYCS must remain simple, portable and independent.

The purpose of governance is not to restrict contributors. The purpose of governance is to preserve the experiment while allowing contributors to improve it.

Contributors are encouraged to challenge implementations.

Contributors should not unintentionally change the hypothesis being tested.

---

# Core Principle

HYCS exists to answer a question:

> What happens when AI becomes the primary maintainer of generated applications?

Every contribution should move the experiment forward.

---

# Project Identity

HYCS is the project.

HYCS is the product users interact with.

HYCS is the platform contributors improve.

HYCS is the experiment being tested.

There is no separate public product name at this stage.

To avoid confusion, contributors should refer to the project as HYCS across the interface, documentation, repository, issues and discussions.

# Framework Policy

HYCS developers may use frameworks.

HYCS-generated applications may not use frameworks.

Contributors are welcome to use framework expertise to improve HYCS.

The experiment concerns generated output, not the tools used to build HYCS.

## Allowed in HYCS' development

* React
* TanStack
* Tailwind
* Vite
* Other Supporting libraries

## Not allowed in HYCS-generated output

* React applications
* Vue applications
* Angular applications
* Svelte applications
* Framework-dependent builds
* Bundler-required output

Generated applications must remain deployable as standalone HTML, CSS and JavaScript.

# Contribution Philosophy

Contributors should solve problems.

Contributors should not redesign the project without discussion.

A contribution should answer:

1. What problem exists?
2. Why does it matter?
3. What changed?
4. How was it tested?
5. What tradeoffs were introduced?

---

# Documentation Requirement

Every meaningful contribution should leave documentation behind.

Code explains implementation.

Documentation explains intent.

Examples:

* Research findings
* Design decisions
* Architecture notes
* Benchmark results
* Integration guides

Future contributors should understand not only what was built, but why.

# Issue Workflow

Issue
→ Discussion
→ Solution Proposal
→ Implementation
→ Review
→ Merge
→ Documentation

The objective is clarity before code.

---

# Maintainer Responsibilities

Maintainers are responsible for:

* Preserving the experiment
* Protecting project anchors
* Reviewing major changes
* Encouraging contributor growth
* Maintaining documentation quality

Maintainers should enable contributors, not gatekeep them.

---

# Decision Rule

When uncertainty exists, ask:

> Does this contribution help us better understand or improve the HYCS experiment?

If the answer is yes, it likely belongs in the project.

If the answer is no, reconsider whether the contribution aligns with the purpose of HYCS.
